3x slab-out-of-bounds in fotg210 UDC via attacker-controlled wIndex

Bounds-check omission in drivers/usb/fotg210/fotg210-udc.c leads to three independently reachable out-of-bounds accesses of fotg210_udc.ep[].

The endpoint array is declared with FOTG210_MAX_NUM_EP = 5 (valid indices 0–4), but request handlers derive an endpoint number from: 

ctrl->wIndex & USB_ENDPOINT_NUMBER_MASK

This produces values 0–15 with no bounds validation before indexing fotg210->ep[epnum].

A malicious USB host can send a crafted SETUP packet with wIndex >= 9, triggering slab-out-of-bounds reads beyond the fotg210_udc allocation and immediate dereference in multiple execution paths.

All three vulnerabilities are independently confirmed by KASAN slab-out-of-bounds reports on kernel 7.0.0-rc7+ via PoC modules:

  • poc1_clear_feature.ko
  • poc2_set_feature.ko
  • poc3_get_status.ko

1. Environment

Kernel    : Linux 7.0.0-rc7+
Arch      : x86_64 (QEMU i440FX)
KASAN     : enabled (generic + inline)
KASLR     : disabled (nokaslr)

2. Threat Model

USB device controller runs on the peripheral side. Attacker is a malicious USB host. No authentication required. Attack surface is USB SETUP packet processing.

3. Root Cause

Structure: 

#define FOTG210_MAX_NUM_EP 5
struct fotg210_ep *ep[5];

But index is derived from: 

USB_ENDPOINT_NUMBER_MASK = 0x0f   // 0–15

No bounds validation exists before indexing.

Heap layout confirms ep[] sits near end of struct: 

ep[5] @ offset 952 → kmalloc-1k boundary at 1024

Thus:

  • 0–4: valid
  • 5–8: intra-object corruption
  • 9–15: slab OOB

4. Vulnerabilities

Vuln 1 — clear_feature OOB dereference

ep = fotg210->ep[ctrl->wIndex & USB_ENDPOINT_NUMBER_MASK];
ep->wedged

Out-of-bounds pointer is dereferenced immediately, leading to crash or struct reinterpretation of adjacent heap memory.

Vuln 2 — set_feature MMIO corruption chain

ep = fotg210->ep[epnum];
fotg210_set_epnstall(ep);

OOB pointer fields (fotg210, dir_in, epnum) are used to compute MMIO register addresses: 

iowrite32(value, reg);

Heap corruption directly influences hardware register access.

Vuln 3 — get_status information leak path

fotg210_is_epnstall(fotg210->ep[epnum]);

OOB-derived fields are used in: 

ioread32(reg);

Result propagates into USB response via ep0_data.

5. KASAN Summary

BUG: slab-out-of-bounds in fotg210_* functions
Read past kmalloc-1k boundary
Offset overflow ~120 bytes beyond allocated region

6. Impact

  • Kernel slab OOB read
  • Cross-object heap interpretation
  • MMIO address manipulation via corrupted struct fields
  • Information leak via USB control response

7. Fix

u8 epnum = ctrl->wIndex & USB_ENDPOINT_NUMBER_MASK;

if (epnum >= FOTG210_MAX_NUM_EP) {
    fotg210_request_error(fotg210);
    return;
}

struct fotg210_ep *ep = fotg210->ep[epnum];

All accesses must validate before dereferencing.
