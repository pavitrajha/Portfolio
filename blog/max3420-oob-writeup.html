MAX3420 USB Device Controller: Endpoint Index Out-of-Bounds via wIndex

This report describes an out-of-bounds read and write condition in the MAX3420 USB Device Controller driver. The issue stems from using a host-controlled wIndex field from USB SETUP packets as an array index without validating it against the endpoint table size.

Because the driver operates in USB device mode, all control requests originate from an untrusted USB host. Fields in SETUP packets must therefore be treated as attacker-controlled inputs.

Overview

The driver derives an endpoint index directly from the USB request: 

id = udc->setup.wIndex & USB_ENDPOINT_NUMBER_MASK;

where:

USB_ENDPOINT_NUMBER_MASK = 0x0f  // range 0–15
MAX3420_MAX_EPS = 4             // valid range 0–3

No validation is performed to ensure the derived index is within bounds before indexing: 

udc->ep[id]

This creates a direct out-of-bounds access into a fixed-size kernel array.

Attack Surface

USB SETUP packets are received from hardware via SPI FIFO registers and passed into driver state without sanitization: 

spi_rd_buf(udc, MAX3420_REG_SUDFIFO, &setup, 8);

udc->setup = setup;
udc->setup.wIndex = cpu_to_le16(setup.wIndex);

The wIndex field is later used in control-flow and memory access decisions.

Vulnerable Access Pattern

The endpoint pointer is resolved directly from the unchecked index: 

ep = &udc->ep[udc->setup.wIndex & USB_ENDPOINT_NUMBER_MASK];

Because MAX3420_MAX_EPS = 4, any value ≥ 4 results in out-of-bounds access.

Vulnerability 1: max3420_getstatus() OOB Read

In max3420_getstatus(), the endpoint structure is dereferenced directly: 

ep = &udc->ep[udc->setup.wIndex & USB_ENDPOINT_NUMBER_MASK];

If the derived index exceeds bounds, ep points outside the array. Subsequent field reads operate on invalid memory.

This results in a kernel heap out-of-bounds read when accessing endpoint state fields.

Vulnerability 2: max3420_set_clear_feature() OOB Write

In max3420_set_clear_feature(), the same unchecked index is used: 

id = udc->setup.wIndex & USB_ENDPOINT_NUMBER_MASK;
ep = &udc->ep[id];

The returned pointer is later used for state modification operations on the endpoint structure.

If id ≥ MAX3420_MAX_EPS, writes occur outside the bounds of the endpoint array, corrupting adjacent kernel memory.

Memory Layout Impact

The endpoint table is embedded in a fixed-size structure: 

struct max3420_ep ep[4];

Consequences of out-of-bounds indexing:

  • Indices 0–3: valid endpoint objects
  • Indices ≥ 4: adjacent fields within struct max3420_udc
  • Further indices: spill into surrounding heap allocation

Because the array is contiguous in memory, OOB access directly overlaps unrelated driver state.

Trigger Condition

A malformed USB control request can trigger the issue: 

bmRequestType = 0x02
bRequest      = SET_FEATURE
wValue        = ENDPOINT_HALT
wIndex        = 0x0007
wLength       = 0x0000

Here, wIndex = 7 produces an endpoint index outside the valid range [0–3].

Execution Flow

The vulnerable path is reachable through standard USB interrupt handling: 

max3420_handle_irqs()
    → SUDAVIRQ detected
        → max3420_handle_setup()
            → SPI read from SUDFIFO
            → max3420_getstatus()
               OR max3420_set_clear_feature()
                   → udc->ep[index] (unchecked)

No validation occurs between packet reception and array access.

Crash Evidence

AddressSanitizer report (userspace harness)

runtime error: index 7 out of bounds for type 'max3420_ep [4]'
ERROR: AddressSanitizer: heap-buffer-overflow

WRITE of size 1 at 0x... (168 bytes past region)

#0 max3420_set_clear_feature
#1 run_test
#2 main

The failure occurs because the computed index (7) exceeds the allocated array size (4), leading to an out-of-bounds write into adjacent memory.

Root Cause

The vulnerability arises from treating a masked USB field as a trusted bounded index: 

wIndex & USB_ENDPOINT_NUMBER_MASK

The mask reduces values to 0–15, but no secondary validation enforces the actual array bounds.

This disconnect between protocol encoding and memory layout results in unsafe indexing.

Fix

Bounds validation must occur before array access: 

id = udc->setup.wIndex & USB_ENDPOINT_NUMBER_MASK;

if (id >= MAX3420_MAX_EPS)
    goto stall;

ep = &udc->ep[id];

This check must be applied consistently across all endpoint-accessing functions.

Affected Scope

All kernels containing the MAX3420 USB Device Controller driver are affected.
