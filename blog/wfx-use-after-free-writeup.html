Silicon Labs wfx wireless driver: Invalid post-lifetime object access via uncancelled work items

Silicon Labs wfx wireless driver | Linux kernel | Race condition / Use-After-Free in interface teardown

Summary

An invalid post-lifetime object access vulnerability exists in the drivers/net/wireless/silabs/wfx/ driver. The flaw is caused by asynchronous work items embedded within struct wfx_vif that remain active and queued after mac80211 zero-initializes the instance memory sequence following an interface removal call.

When the uncancelled work queues execute against the cleared driver private region, fields inside the structure are dereferenced as null or corrupt pointers, leading to arbitrary memory accesses or kernel panic conditions.

Vulnerability details

During interface teardown, wfx_remove_interface() clears out specific async handlers but omits others. Specifically, beacon_loss_work is properly synchronized, while update_tim_work and tx_policy_upload_work are ignored: 

cancel_delayed_work_sync(&wvif->beacon_loss_work); /* Only item canceled */
wdev->vif[wvif->id] = NULL;
mutex_unlock(&wdev->conf_mutex);

Once drv_remove_interface() executes and returns back to the subsystem stack, mac80211 directly forces a memset block to clear the underlying private block structure tracking the virtual interface data (net/mac80211/iface.c): 

drv_remove_interface(local, sdata);
memset(sdata->vif.drv_priv, 0, local->hw.vif_data_size); /* wvif memory zeroed out */

Any pending or execution-deferred work item firing past this absolute memory clearance point processes zeroed wvif fields natively.

Root cause

The driver lacks complete serialization barriers during its teardown lifecycle path. A quick inspection confirms that no synchronization hooks exist across the codebase to capture these specific asynchronous elements before clearing the interface state context: 

grep -rn "cancel_work_sync.*update_tim\|cancel_work_sync.*tx_policy" drivers/net/wireless/silabs/wfx/
(no matches)

The uncancelled callbacks are broken down below:

update_tim_work (sta.c)

static void wfx_update_tim_work(struct work_struct *work)
{
    struct wfx_vif *wvif = container_of(work, struct wfx_vif, update_tim_work);
    wfx_update_tim(wvif); /* Operates on zeroed memory with no mutex alignment */
}

tx_policy_upload_work (data_tx.c)

The execution handler unrolls structures under a local per-vif spinlock block (cache->lock), processing memory references directly out of the zeroed object allocation frame.

Race window

The race condition triggers inside the core wireless state transition machine via ieee80211_do_stop(): 

sta_info_flush()
  -> __sta_info_destroy_part2()
    -> __sta_info_recalc_tim(sta, true)  /* sta->dead evaluation evaluates to false */
      -> drv_set_tim()
        -> wfx_set_tim()
          -> schedule_work(&wvif->update_tim_work)  /* Asynchronous work item queued here */

[wiphy_work_cancel() clears mac80211 sub-allocations exclusively]

drv_remove_interface()
memset(sdata->vif.drv_priv, 0, ...)  /* Private driver context zeroed out immediately */

[Asynchronous workqueue execution wakes up -> dereferences nullified memory space]

Impact

  • Kernel panic state due to null-pointer/invalid address dereferencing
  • Use-After-Free (UAF) access window over recycled structures
  • Race vulnerability exploitation within interface hot-unplug paths

Fix

Introduce explicit cancellation steps for the omitted work structures inside the teardown pipeline sequence before state invalidation: 

  cancel_delayed_work_sync(&wvif->beacon_loss_work);
+ cancel_work_sync(&wvif->update_tim_work);
+ cancel_work_sync(&wvif->tx_policy_upload_work);
  wdev->vif[wvif->id] = NULL;

Affected versions

Identified in mainline development trees (Linux 7.0-rc7 architecture) and upstream distributions matching CONFIG_WFX=y/m settings without independent patch implementations as of May 2026.

Notes

This structural vulnerability represents a classic teardown sequencing bug frequently found across multi-layered wireless driver designs where global subsystem data layers clear resources before driver-managed background worker tasks have finished processing.

Upstream Patch & Timeline

The vulnerability was reported upstream through the Linux Kernel Mailing List (LKML), followed by patch review, maintainer discussion, and integration into the upstream kernel tree.
